Введение

Функция Notifications (уведомления) позволяет настраивать автоматические сообщения, которые будут отправляться при возникновении определенных событий на платформе. Данная статья подробно описывает, как настраивать и управлять уведомлениями на вашей платформе.

Обзор уведомлений:

При настройке уведомления необходимо установить:

  • Event type (тип события): Тип события платформы, которое запускает отправку сообщения. Например, User enrolled in a course (пользователь записан на курс).
  • Recipients (получатели): Предполагаемые получатели сообщения. Они определяются в зависимости от их отношения к событию. Например, вы можете отправить уведомление User enrolled in a course (пользователь записан на курс) учащемуся, который был записан, или инструктору (Instructor) соответствующего курса.
  • Delivery channel (канал доставки): Способ отправки сообщения. Например, email (электронная почта) или in-platform notification (уведомление внутри платформы).
  • Message (сообщение): Фактический текст отправляемого сообщения. Обычно его следует адаптировать к выбранному типу события, а также к предполагаемым получателям.
  • Timing (время): В зависимости от типа события у вас могут быть некоторые варианты планирования времени отправки сообщения. Например, немедленно во время события или через определенное количество дней после события.
  • Trigger filters (фильтры триггеров): Вы можете применить фильтры, чтобы сузить случаи отправки уведомления. Например, вы можете создать уведомление User enrolled in a course (пользователь записан на курс), которое отправляется только для определенного набора курсов или только когда определенная группа пользователей записывается на курс.

Статья Notification events and conditions (события и условия уведомлений) предоставляет полный список событий платформы, для которых вы можете настроить уведомление. Для каждого типа события она также показывает группы получателей, которым вы можете направить уведомление, и возможные варианты планирования.

Предварительные требования

Права доступа (Permissions)

Power Users (опытные пользователи) должны получить соответствующие разрешения на уведомления, чтобы иметь возможность просматривать и управлять уведомлениями. И они будут иметь видимость только тех уведомлений, которые касаются их назначенных ресурсов (группы, филиалы, курсы и планы обучения).

Просмотр списка уведомлений

Меню Admin > Notifications > Manage

Выберите Admin menu > Notifications > Manage для доступа к странице управления уведомлениями. Здесь вкладка Notifications показывает список уже настроенных уведомлений.

Страница управления уведомлениями со списком настроенных уведомлений

Для каждого уведомления вы можете видеть его Event type (тип события) и иконки, представляющие настроенные каналы доставки, а также его Status (статус). Только активные (Active) уведомления могут быть отправлены. Некоторые уведомления имеют иконку Play (треугольник), которую можно использовать для ручного запуска отправки. Вы можете применить фильтры или использовать поле поиска, чтобы найти конкретное уведомление.

Доступные фильтры: Event category (категория события), Event type (тип события), Attribution (атрибуция, только для Community уведомлений) и Status (статус: Active, Inactive или оба).

Чтобы увидеть больше деталей о конфигурации уведомления, нужно либо нажать на его название, либо выбрать меню с тремя точками > Edit (редактировать).

Совет: В этом списке Power Users смогут видеть только уведомления, относящиеся к их назначенным ресурсам. Это означает, что любые группы, филиалы или курсы/планы обучения, установленные как фильтры триггеров для уведомления, также должны быть назначены Power User в качестве ресурса. В противном случае уведомление не будет видно.

Вкладка Logs (логи) позволяет проверить, какие уведомления были недавно отправлены, автоматически или вручную. См. главу Проверка логов уведомлений.

Создание нового уведомления

Чтобы создать новое уведомление:

  1. 1Выберите Admin menu > Notifications > Manage и нажмите иконку + в верхнем правом углу.
  2. 2В открывшейся панели Create notification (создать уведомление) определите Event (событие) платформы, для которого вы хотите настроить это уведомление: сначала выберите Event category (категорию события) (например, Courses), а затем выберите конкретный Event type (тип события) в этой категории (например, ILT event starting).
  3. 3Дайте уведомлению описательное Name (имя), которое поможет вам легко идентифицировать его в списке и логах.
    → По умолчанию это поле заполняется типом события, но помните, что у вас может быть несколько уведомлений для одного типа события (например, нацеленных на разных получателей или разные наборы курсов). Поэтому полезно предоставить отличительное имя.
  4. 4Когда будете готовы, нажмите Create and edit (создать и редактировать), чтобы продолжить настройку.
    Обратите внимание, что когда вы это делаете, выбранное событие становится доступным только для чтения. Вы не можете позже редактировать уведомление, чтобы изменить его событие, однако все другие аспекты его конфигурации могут быть впоследствии изменены.
  5. 5Продолжите, как описано в главе Настройка уведомления.
Совет: Вы также можете создать новое уведомление, начав с существующего. Для этого из основного списка уведомлений откройте меню с тремя точками рядом с уведомлением и выберите Duplicate (дублировать).

Тип события копируется из исходного уведомления и доступен только для чтения. Затем вы можете установить имя и выбрать, какие другие части его конфигурации копировать: Timing (время), Trigger filters (фильтры триггеров), Recipients (получатели), Delivery channels (каналы доставки).

Это удобный вариант, например, если вы хотите сделать несколько копий одного уведомления для отправки разным группам получателей.

Настройка уведомления

Чтобы начать настройку нового уведомления, сначала выполните шаги из главы Создание нового уведомления.

Чтобы получить доступ к странице конфигурации существующего уведомления, выберите Admin menu > Notifications > Manage и нажмите на Name (имя) уведомления.

Страница настройки уведомления, показывающая вкладки Properties, Recipients и Trigger filters

На странице конфигурации уведомления вы увидите три вкладки: Properties (свойства), Recipients (получатели) и Trigger filters (фильтры триггеров). Здесь вы можете настроить все параметры уведомления, кроме его события, которое доступно только для чтения.

Действуйте следующим образом:

  • Вкладка Recipients: Установите, кому адресовано уведомление. См. главу Установка получателей уведомления.
  • Вкладка Properties > Composition: Здесь вы можете установить один или несколько каналов доставки и составить текст отправляемого сообщения. См. главу Составление сообщения
  • Вкладка Properties > Details and timing: Здесь при желании вы можете редактировать имя уведомления. Помните, что событие уведомления доступно только для чтения. Затем в области Scheduler настройте время отправки уведомления. См. главу Планирование уведомления.
  • Вкладка Triggers and filters: Здесь вы можете установить, какие ресурсы (группы пользователей/филиалы, курсы и планы обучения) будут запускать уведомление. См. главу Установка фильтров триггеров для уведомления.

В любой момент вы всегда можете нажать Save changes (сохранить изменения), чтобы сохранить частичную конфигурацию.

Установка получателей уведомления

На странице конфигурации уведомления выберите вкладку Recipients.

Здесь вы можете выбрать, какие "роли" пользователей будут получать уведомление, основываясь на их отношении к событию уведомления. Опции могут включать: Users (пользователи), Power Users (опытные пользователи), Managers (менеджеры), Instructors (инструкторы), Superadmins (суперадминистраторы).

Например, вы можете настроить уведомление Certification has expired (сертификация истекла), адресованное непосредственно затронутым пользователям, или адресованное менеджерам этих пользователей.

Доступные для выбора роли зависят от типа события. Статья Notification events and conditions (события и условия уведомлений) подробно описывает, какие получатели разрешены для каждого события уведомления. Некоторые события уведомлений могут не требовать установки роли получателя.

Обратите внимание, что вы не можете выбрать несколько ролей получателей. Для этого вам потребуется настроить два отдельных уведомления с одинаковым типом события, но с разными получателями.

→ При этом помните, что некоторые роли получателей могут пересекаться. Например, один и тот же человек может быть как Power User, так и Manager, и поэтому будет получать уведомления, адресованные обеим ролям.
Выбранная цель Описание
Users (пользователи) Пользователи, затронутые событием, получают уведомление.

Только для уведомления New ILT session created (создана новая ILT сессия) вы можете дополнительно выбрать между "all users (все пользователи)" или пользовательским выбором, который нацелен на пользователей, не записанных ни в одну сессию, и/или пользователей в списке ожидания как минимум одной сессии.
Power Users (опытные пользователи) Power Users, чьи назначенные ресурсы затронуты событием, получают уведомление.

При выборе этой опции вы можете установить "All Power User profiles (все профили Power User)" или указать конкретный профиль.

Примечание: Роль получателя Power User может быть установлена только Superadmin. Когда Power Users настраивают уведомления, они не могут установить эту роль получателя, и если она уже установлена, они не могут ее изменить.
Managers (менеджеры) Менеджеры, члены команды которых затронуты событием, получают уведомление.
Instructors (инструкторы) Инструкторы курсов, затронутых событием, получают уведомление.

Примечание: Это применяется к пользователям, которые записаны в e-learning курс с уровнем записи Instructor.

→ В случае ILT курсов инструкторы назначаются на курс или сессию, но не записываются в них. Поэтому, если у вас есть уведомление типа "ILT session starting" с инструкторами в качестве получателей, оно НЕ будет отправлено инструкторам, если они также не записаны в курс/сессию как учащиеся.
Superadmins (суперадминистраторы) Все суперадминистраторы, определенные на платформе, получают уведомление
Superadmin плюс одна другая роль Эта опция доступна только для уведомления Subscription plan has been purchased (план подписки был приобретен). Она позволяет установить в качестве получателей как суперадминистраторов, так и любую из других ролей.

Составление сообщения уведомления

На странице конфигурации уведомления выберите вкладка Properties > Composition.

В разделе Composition вы увидите один или несколько заголовков, соответствующих каналам доставки, включенным на вашей платформе. Они могут включать: Email (электронная почта), Platform notification (уведомление платформы), mobile push notification (мобильное push-уведомление).

Вы должны настроить как минимум один канал доставки, чтобы иметь возможность активировать уведомление.

Интерфейс составления уведомления с несколькими каналами доставки

Email уведомление

Для email уведомления выберите вкладка Properties > Composition > Email. Затем в панели справа отметьте флажок Enable email as delivery channel (включить email как канал доставки).

В области Sender (отправитель) вам нужно будет указать имя отправителя и email адрес.

Обратите внимание: Email адрес, который вы вводите для отправителя, например noreply@mydomain.com, должен быть предварительно настроен в Domain management (управлении доменами). См. статью configuring email sender domains (настройка доменов отправителей email).

Затем продолжите, как указано в введите текст сообщения

In-platform уведомление

Platform notifications (уведомления платформы) доставляются пользователям в области Notifications платформы (иконка колокольчика в заголовке).

Чтобы настроить этот канал доставки, выберите вкладка Properties > Composition > Platform notification, и в панели справа отметьте флажок Enable platform notification as delivery channel (включить платформенное уведомление как канал доставки).

Затем продолжите, как указано в введите текст сообщения

Введите текст сообщения

В панели, соответствующей включенному методу доставки (Email или Platform notification), вы найдете область Message (сообщение), в которой можете составить содержимое вашего уведомления на одном или нескольких языках.

Язык по умолчанию

Прежде всего, введите Subject (тему) и Body (тело) вашего сообщения на языке по умолчанию вашей платформы. Язык по умолчанию выбирается автоматически, и если опция Translate fields активна, он будет отмечен как (default). Вы должны ввести текст хотя бы на этом языке по умолчанию, чтобы иметь возможность сохранить уведомление.

→ Для некоторых событий уведомлений эти поля могут быть предварительно заполнены шаблонным текстом, который вы можете настроить.

Для уведомлений, настроенных при входе на платформу расширенного предприятия, текст должен быть введен на языке по умолчанию, установленном для этого клиента расширенного предприятия, который может отличаться от языка по умолчанию основной платформы. В любом случае требуемый язык по умолчанию будет отмечен как (default).

Форматирование

Текст в теле сообщения также может быть отформатирован:

  • Используя кнопки панели инструментов для уведомлений внутри платформы и email. В дополнение к стандартным опциям форматирования текста вы можете вставлять ссылки, изображения и таблицы. Вы также можете войти в режим исходного кода, чтобы редактировать html сообщения напрямую.
  • Для slack уведомлений нет панели инструментов, но вы можете использовать markdown для форматирования текста.

Shortcodes (короткие коды)

Вы можете включить в свое сообщение shortcodes, перечисленные внизу страницы.

Это теги типа {{course_name}} или {{first_name}}, которые будут заменены соответствующими данными - из события уведомления и получателя - при отправке уведомления.

Доступные shortcodes зависят от выбранного события уведомления, и вы можете использовать только те, которые перечислены.

Нажмите на чип shortcode, чтобы вставить его в текущую позицию курсора в теме или теле сообщения.

Для получения дополнительной информации см. статью Shortcodes in notifications (короткие коды в уведомлениях).
Обратите внимание: Если язык платформы по умолчанию изменен в инструменте локализации, то существующие уведомления, в которых отсутствует текст сообщения на новом языке по умолчанию, могут не отправляться*.

При редактировании этих уведомлений вам придется заполнить текст сообщения на новом языке по умолчанию, прежде чем вы сможете сохранить любые изменения в них.

* Язык, используемый для отправки уведомления, определяется выбором первого из доступных: предпочтительный язык пользователя-получателя > язык по умолчанию платформы расширенного предприятия получателя (если применимо) > язык по умолчанию родительской (основной) платформы > английский.

И если ни один из них недоступен, уведомление не будет отправлено.

Другие языки

Если вы хотите ввести переводы сообщения уведомления на другие языки, включите опцию Translate fields (переводить поля). Затем из выпадающего списка Languages (языки) выберите желаемый язык и составьте и отформатируйте текст темы и тела сообщения на этом языке.

→ Повторите эти шаги, пока не составите сообщение уведомления для всех необходимых языков.

Уведомления, составленные на нескольких языках, будут доставляться получателям на основе их индивидуальных языковых предпочтений.

Вложения (Attachments)

В области Attachments внизу вы можете включить календарное вложение в уведомление. Эта опция доступна только для определенных типов событий (относящихся к ILT сессиям и курсам) и только для каналов доставки email или Slack.

Mobile push уведомление

Mobile push notifications (мобильные push-уведомления) доставляются пользователям приложения Go.learn или брендированного мобильного приложения, при условии, что они включили это на своем устройстве.

Обратите внимание, что канал доставки mobile push доступен только для подмножества типов событий. Вы увидите опцию Mobile push notification только для следующих поддерживаемых типов событий:

Course (курс): User approval, User approved into ILT session, User enrolled in ILT session, User enrolled in a course, ILT session starting, Learner completed a course, ILT event starting, ILT session changed, ILT session deleted, Learner has yet to complete a course, Assignment has been evaluated, Instructor evaluated a test question, New ILT session created, User was declined from an ILT session

Learning plans (планы обучения): User has been enrolled in a learning plan, Learner has completed a learning plan, Learning plan enrollment expiring, New content in learning plan, User was unenrolled from a learning plan

Чтобы настроить канал доставки mobile push, выберите вкладка Properties > Composition > Mobile push notification, и в панели справа отметьте флажок Enable mobile push notification as delivery channel (включить мобильное push-уведомление как канал доставки).

Теперь вы можете ввести Title (заголовок) (максимум 50 символов) и простой текст Message (сообщение) (максимум 150 символов).

Вы можете включать shortcodes только в текст сообщения. Учтите, что текст должен оставаться в пределах 150 символов даже после замены shortcodes их значениями. В противном случае любой избыточный текст будет обрезан в уведомлении, которое получают пользователи.

Раздел App URL внизу показывает ссылку мобильного приложения, которую откроет уведомление. Это поле доступно только для чтения и автоматически заполняется на основе типа события.

Как и для других каналов уведомлений, вы можете включить опцию Translate fields, чтобы ввести сообщение мобильного push-уведомления на других языках.

Совет: В зависимости от настроек мобильного устройства получателя, если несколько похожих уведомлений поступают за короткое время, они могут быть сгруппированы или свернуты в одно. Например, если (не-digest) уведомление User enrolled in a course настроено для отправки Power Users, и 4 пользователя записываются на курс, Power User может получить только одно сообщение, а не 4 отдельных, как ожидалось.

→ Этого можно избежать, вставив shortcodes, такие как имя записанного пользователя, чтобы убедиться, что все сообщения различны.

Планирование уведомления

На странице конфигурации уведомления выберите вкладка Properties > Details and timing. В области Scheduler (планировщик) вы увидите доступные опции времени для выбранного события уведомления.

Опции времени для каждого типа события перечислены в статье Notification events and conditions (события и условия уведомлений).

В зависимости от типа события может быть только одна фиксированная опция планирования, или вам может быть предложен выбор между двумя или более из следующих: At the time of the event (во время события), Before the event (до события), After the event (после события), Manually (вручную) и Every (каждый).

Для многих опций планирования вы можете установить точное Time (время) дня, когда хотите отправить уведомление, а также указать соответствующий Time zone (часовой пояс).

Советы: Время должно быть введено в формате hh:mm am/pm. Например 08:15 am.

→ Обратите внимание, что в формате am/pm 12:00 am соответствует 00:00 в 24-часовом времени (полночь, начало дня), а 12:00 pm соответствует 12:00 в 24-часовом времени (полдень, середина дня).

В планировщике, когда вы указываете временные рамки, разрешенные диапазоны следующие: 1→72 часа, 1→365 дней, 1→52 недели, 1→24 месяца.
Интерфейс планировщика уведомлений с опциями настройки времени

At the time of the event (во время события)

Уведомление будет отправлено получателям, как только запускающее действие будет завершено на платформе. Например, в случае события User bought a course (пользователь купил курс), с этой настройкой уведомление будет отправлено, как только пользователь завершит покупку.

Обратите внимание: В некоторых случаях уведомление "во время" может быть несколько задержано относительно события. Например, система мониторит чек-листы на обновления каждые три часа, так что уведомление чек-листа может быть задержано до трех часов от фактического события.

Before the event (до события)

Эта опция доступна для событий, которые должны произойти в известное время. Уведомление будет отправлено получателям за определенное время до события, например, до начала вебинара или ILT сессии.

Вы можете установить, за сколько часов, дней, недель или месяцев заранее будет отправлено уведомление.

Если вы устанавливаете дни, недели или месяцы, вы также можете запланировать время дня, когда хотите отправить уведомление, и указать соответствующий часовой пояс.

Часы до события:

Планировщик отправит уведомление за указанное количество часов до события, также учитывая минуты. Например, предположим, что ILT сессия должна начаться в 10:25:

  • Если вы установите уведомление за два часа до события, оно будет отправлено в 8:25.
Совет: Планировщик проверяет события в начале каждого часа, поэтому не будет работать как ожидается, если вы:
  • Создаете уведомление в течение часа, когда оно должно быть отправлено. В приведенном выше примере, если вы создадите уведомление в 8:05, оно не будет отправлено (в 8:00 система уже проверила события в промежутке времени между 10-11)
  • Изменяете время события в течение часа, когда должно было быть отправлено исходное уведомление. В приведенном выше примере вам нужно было бы изменить время ILT сессии до 8:00.

Дни до события

Для дней до события каждый день в запланированное время система будет искать события, происходящие в течение 24-часового промежутка времени, который находится на указанное количество дней в будущем.

Например, если вы установите 2 дня до события, временной промежуток составляет от 48 до 72 часов в будущем.

Нижний лимит включен, а верхний лимит исключен. Так в этом примере:

48hrs =< время до события < 72hrs

Часы до события считаются от времени, которое вы установили для отправки уведомления.

Совет: "Дни до события", которые вы устанавливаете, могут не соответствовать календарным дням.

Предположим, ILT сессия начинается в среду в 7:00, и вы настроили уведомление за 1 день до события, отправляемое в 8:00. Один день до события означает от 24 до 48 часов:

24hrs =< время до события < 48hrs

→ В понедельник в 8:00 событие находится за 47 часов, поэтому уведомление будет отправлено. Даже несмотря на то, что понедельник - это два календарных дня до события.

→ Если время уведомления было бы установлено на 7:00, то уведомление было бы отправлено во вторник (за один календарный день до события).

Вообще говоря, "дни до события" соответствуют календарным дням, если время уведомления раньше (или равно) времени события. В противном случае уведомление будет отправлено на один "дополнительный" календарный день заранее.

Недели до события

Для недель до события логика такая же, как для дней до, с количеством дней = количество недель * 7.

Например, для одной недели до события каждый день в запланированное время система будет искать события в 24-часовом промежутке времени между 7 днями (168 часов) и 8 днями (192 часа) в будущем.

Совет: Уведомление не обязательно будет отправлено в тот же день недели, что и событие. Например, если событие в среду в 10:00, и вы установите уведомление за 1 неделю до в 9:00, то уведомление будет отправлено в предыдущую среду. Но если вы установите уведомление на 11:00, оно будет отправлено на день раньше, во вторник.

Месяцы до события

Для месяцев до события каждый день в запланированное время система будет искать события в 24-часовом промежутке времени, который находится на указанное количество месяцев в будущем.

Пример: Есть уведомление, запланированное за один месяц до события, в 9:00. Каждый день в запланированное время система просматривает текущую дату (например, 2 марта, 9:00) и увеличивает месяц на один (например, 2 апреля, 9:00): Затем она ищет события в 24-часовом промежутке времени между 2 апреля, 9:00 и 3 апреля, 8:59:59.

Обратите внимание: 31 марта, чтобы увеличить месяц на один, система возьмет 30 апреля (т.е. последний день этого месяца), и поэтому будет искать события в том же промежутке времени, который она рассматривала 30 марта. Поэтому в случае событий, происходящих в последние дни месяца, уведомление может быть отправлено два дня подряд.

After the event (после события)

Уведомление будет отправлено получателям через определенное время после завершения события или действия, например, после того, как учащийся завершил курс.

Вы можете установить, через сколько часов, дней, недель или месяцев после события будет отправлено уведомление. Если вы устанавливаете дни, недели или месяцы, вы также можете установить время дня (и часовой пояс), когда хотите его отправить.

Часы после события

Если вы используете опцию часов, уведомление будет задержано на точное количество часов, которое вы указываете от события, также учитывая минуты.

Дни после события

Обратите внимание, что для дней после события задержка вычисляется следующим образом:

  • Переместиться вперед на указанное количество дней (24-часовых периодов) от даты и времени события.
  • Затем отправить уведомление в следующее первое появление запланированного времени, которое вы установили в поле времени.
→ Исключением из логики, описанной в этом разделе, является уведомление Learner has yet to complete a course (учащийся еще не завершил курс), которое вычисляет задержку на основе "человеческих дней". Поэтому от даты события оно перемещается вперед на указанное количество дней, и в этот день отправляет уведомление в запланированное время.
Совет: Если время уведомления раньше времени события, уведомление будет отправлено на день позже, чем вы ожидаете. Например, предположим, событие происходит в понедельник в 15:00, и вы настраиваете уведомление через 4 дня после события в 10:00:
  • 4 дня после события - это пятница в 15:00
  • но следующее появление запланированного времени - суббота в 10:00, и это когда отправляется уведомление.

Недели после события

Опция недель после события следует аналогичной логике дням после события. От даты и времени события планировщик перемещается вперед на задержку = (количество недель) * (7дней/неделя) * (24 часа/день). А затем отправляет уведомление в следующее первое появление запланированного времени.

Совет: Уведомление не обязательно будет отправлено в тот же день недели, что и событие. Например, если событие в понедельник в 11:00, и вы установите уведомление через 1 неделю после в 9:00, то уведомление будет отправлено во вторник следующей недели. Этот дополнительный день возникает, если время уведомления раньше времени события.

Месяцы после события

В случае месяцев после события планировщик перемещается вперед на указанное количество месяцев, а затем отправляет уведомление в следующее первое появление запланированного времени.

Например:

  • Для события 3 июня в 8:00, четыре месяца после события - это 3 октября в 8:00.
  • Однако для события 31 марта один месяц после события - это 30 апреля (если месяц короче, планировщик все равно берет последний день)

Как упоминалось для дней и недель после события, уведомление будет отправлено на "дополнительный" день позже, если время уведомления раньше времени события.

Manually (вручную)

Некоторые уведомления могут быть установлены для отправки Manually. Это означает, что они никогда не отправляются автоматически планировщиком. Вместо этого они могут быть запущены вручную нажатием иконки "play" в списке уведомлений.

Every (каждый)

Это опция, доступная для digest уведомлений, и позволяет отправлять уведомление на повторяющейся основе: ежечасно, ежедневно, еженедельно или ежемесячно.

Для получения дополнительной информации см. главу о Digest уведомлениях.

Примечания о планировании уведомлений

  • Невозможно повторно отправить запланированные уведомления.
  • Запланированные уведомления отправляются периодически и не запускаются во время события. Эти уведомления используют задание, которое выполняется по расписанию (частота зависит от конфигурации уведомления).

Установка фильтров триггеров для уведомления

Trigger filters (фильтры триггеров) позволяют указать, для каких пользователей и для какого контента должно отправляться уведомление.

→ Например, вы можете отправлять уведомление Learner has completed a course (учащийся завершил курс) только когда учащиеся определенного филиала завершают курс. Или вы можете отправлять его только для определенных курсов.

Проверка фильтров триггеров

На странице конфигурации уведомления выберите вкладку Trigger filters. Здесь вы можете проверить любые уже настроенные фильтры триггеров, используя вертикальную навигацию под Trigger type (тип триггера).

Интерфейс настройки фильтров триггеров, показывающий опции групп пользователей, филиалов и курсов

Users (пользователи):

Эта часть показывает группы пользователей и/или филиалы пользователей, которые могут запускать это уведомление. Если установлено несколько групп или филиалов, логика между ними OR, что означает, что достаточно пользователю принадлежать хотя бы к одной из них.

→ Когда группы или филиалы не установлены: Для уведомления, настроенного Superadmin, это означает, что все пользователи могут запускать уведомление. Но для уведомления, настроенного Power User, это означает, что никакие пользователи не могут запускать уведомление.

Courses and learning plans (курсы и планы обучения):

Эта часть показывает контент, который может запускать это уведомление.

→ Когда курсы или планы обучения не установлены: Для уведомления, настроенного Superadmin, это означает, что весь контент может запускать уведомление. Но для уведомления, настроенного Power User, это означает, что никакой контент не может запускать уведомление.

Добавление фильтра триггера

Чтобы добавить фильтр триггера, нажмите иконку + в верхнем правом углу. В панели Add trigger filters (добавить фильтры триггеров) выберите тип применяемого фильтра триггера (Groups, Branches или Courses and learning plans) и нажмите Next (далее). Затем сделайте выбор в соответствии с выбранным типом ресурса.

  • Groups (группы): Отметьте флажок рядом с группами пользователей, которые должны запускать уведомление.
  • Branches (филиалы): Отметьте флажок рядом с каждым из филиалов пользователей, которые должны запускать уведомление. Нажмите флажок еще раз, чтобы включить/исключить потомков.
  • Courses and learning plans (курсы и планы обучения): Отметьте флажок рядом с каждым курсом или планом обучения, который должен запускать уведомление.

Если вы Power User, вы сможете видеть здесь только те группы, филиалы или курсы и планы обучения, которые назначены вам как ресурс.

Фильтры триггеров и видимость

Когда Superadmin создает уведомление:

  • Оно применяется по умолчанию ко всем пользователям и ко всем курсам и планам обучения.
  • Это означает, что если фильтры триггеров не применены, уведомление будет отправляться для всех случаев выбранного события.
  • Фильтры триггеров могут быть применены, чтобы ограничить уведомление определенным набором пользователей или подмножеством учебного контента, или и тем, и другим.
  • Уведомление может быть активировано, даже если фильтры триггеров не применены.

Когда Power User создает уведомление:

  • Оно применяется по умолчанию к никаким пользователям и к никаким курсам или планам обучения. Это означает, что в этом начальном состоянии уведомление не будет отправляться для любого случая выбранного события.
  • Фильтры триггеров могут быть применены, чтобы указать, для каких пользователей и для каких курсов и планов обучения должно отправляться уведомление.
  • Как минимум один фильтр триггера должен быть применен, чтобы иметь возможность активировать уведомление в этом случае.

Power User будет иметь видимость уведомления, только если ему назначены все ресурсы, установленные как фильтры триггеров для него. Для деталей об этом см. статью о Power User permissions for notifications (разрешения Power User для уведомлений). Power User потеряет видимость уведомления, даже того, которое он создал сам, если кто-то другой отредактирует его, добавив фильтры триггеров, не назначенные Power User.

Активация уведомления

На странице списка уведомлений столбец Status показывает, активировано ли каждое уведомление или деактивировано.

Чтобы активировать (или деактивировать) уведомление, нажмите на <имя уведомления>, чтобы открыть его страницу конфигурации, затем используйте кнопку в правом нижнем углу, чтобы переключить статус между Active (активно) и Inactive (неактивно).

Примечание: Чтобы иметь возможность активировать уведомление, у него должен быть установлен как минимум один канал доставки.

В дополнение, Power Users должны установить как минимум один фильтр триггера, чтобы иметь возможность активировать уведомление, поскольку отсутствие фильтров триггеров означает "никаких пользователей и никакого контента". Однако Superadmins могут активировать уведомление даже без применения фильтров триггеров, поскольку в их случае это означает "всех пользователей и весь контент".

Редактирование уведомлений

Настоятельно рекомендуется деактивировать уведомление перед редактированием его конфигурации.

Каждое изменение уведомления запустит удаление и пересоздание запланированного задания.

Digest уведомления (сводные)

Digest notifications (сводные уведомления) группируют несколько уведомлений об одном типе события в одно большое уведомление. Например, вместо получения семи отдельных уведомлений о курсах, которые еще не завершены, пользователь получит одно уведомление со списком курсов, которые ему еще предстоит завершить.

Доступные digest уведомления перечислены в статье Notification events and conditions (события и условия уведомлений).

Digest уведомления настраиваются аналогично другим уведомлениям. Когда вы составляете тело сообщения, shortcode для элементов, перечисляемых в digest, должен быть размещен между тегами {{#items}} и {{/items}}. Например, для Digest: User enrolled in a course (сводка: пользователь записан на курс):

Здравствуйте {{first_name}},

Вы были записаны на следующие курсы:

{{#items}}
{{course_name}} {{subscribed_at}}. Этот курс истекает {{expire_at}} и доступен здесь: {{course_link}}
{{/items}}

Наслаждайтесь!
Обратите внимание: Курсы в digest будут перечислены в порядке создания курса.

Опции Manually и Every

При планировании отправки уведомления, в зависимости от типа digest, доступными опциями могут быть Every и/или Manually.

С опцией Manually digest уведомление может быть отправлено вручную и не будет планироваться автоматически.

В случае Every вы можете установить почасовую, ежедневную, еженедельную или ежемесячную частоту отправки digest. Digest будет содержать элементы, относящиеся к этому временному интервалу. Например, Digest: User enrolled in a course, установленный с еженедельной частотой, будет содержать записи прошлой недели, установленный с ежемесячной частотой будет содержать записи прошлого месяца и т.д.

  • Every hour (каждый час): Digest будет отправляться раз в час.
  • Every day (каждый день): Digest будет отправляться раз в день. Вы можете указать время дня и часовой пояс.
  • Every week (каждую неделю): Digest будет отправляться раз в неделю. Вы можете установить день недели (например, среда) и время дня и часовой пояс.
  • Every month (каждый месяц): Digest будет отправляться раз в месяц. Вы можете установить день месяца (например, 10-е число) и время дня и часовой пояс.

    Обратите внимание, что если вы установите 31-е число, в месяцах с меньшим количеством дней уведомление все равно будет отправлено в последний день месяца.

Вкладка Extra settings

Только для уведомления Digest: Course has expired (сводка: курс истек) есть вкладка Extra settings (дополнительные настройки), где вы можете указать, какие курсы включать в digest уведомление. Обратите внимание, что частота отправки digest все еще та, что установлена на вкладке Properties с опцией Every.

Вы можете выбрать включение:

  • Courses expiring between this notification and the next scheduled one (курсы, истекающие между этим уведомлением и следующим запланированным): Например, если digest отправляется каждую неделю, он будет перечислять все курсы, истекающие в неделю между текущим digest и следующим.
  • Courses expiring within a time frame after the notification is sent (курсы, истекающие в течение временного интервала после отправки уведомления): Digest будет перечислять курсы, истекающие в течение указанного количества дней. (например, с 3 digest будет перечислять все курсы, истекающие через 1, 2 или 3 дня).
  • Courses expiring an exact number of days after the notification is sent (курсы, истекающие точно через количество дней после отправки уведомления): Digest будет перечислять курсы, срок действия которых истекает точно через количество дней (например, с 3 digest будет перечислять все курсы, которые истекают ровно через 3 дня).

Community уведомления (сообщества)

Страница управления уведомлениями также показывает любые доступные Community notifications. Эти типы уведомлений не создаются вручную, как другие. Вместо этого они включаются как набор для данного сообщества из области управления сообществами.

Для этого:

  1. 1Выберите Admin menu > COMMUNITIES > Communities hub. Это откроет страницу управления сообществами, перечисляющую все сообщества, настроенные на платформе.
  2. 2Нажмите имя сообщества, чтобы открыть его страницу конфигурации. Затем на вкладке Settings (настройки) прокрутите вниз до раздела Notifications.
  3. 3Здесь вы можете нажать Create community notifications (создать уведомления сообщества), чтобы автоматически создать набор предварительно настроенных уведомлений для этого конкретного сообщества. Если это действие недоступно, это означает, что оно было выполнено ранее, и эти уведомления уже существуют.
Интерфейс управления сообществами с опцией создания уведомлений

На странице управления уведомлениями вы можете найти набор автоматически созданных Community уведомлений, отфильтровав по Event category = Communities и Attribution = <имя сообщества>.

В отличие от других уведомлений, Community уведомления имеют только вкладку Properties. Вы можете активировать/деактивировать их и редактировать составление сообщения. Однако вы не можете удалять или дублировать эти уведомления.

Для получения дополнительной информации см. статью Community notifications (уведомления сообщества).

Советы по уведомлениям

  • 4-байтовые Unicode символы: 4-байтовые Unicode символы не поддерживаются платформой при создании или редактировании уведомления.
  • Длинные уведомления: Уведомления имеют ограничение в 65535 символов. Любое уведомление, которое генерируется и превышает это ограничение, не будет отображаться корректно. Обратите внимание, что это ограничение символов - это ограничение количества символов, получаемых системой уведомлений. Таким образом, контент, который не отображается, такой как HTML код и преобразованные shortcodes, также учитывается в этом ограничении.
  • Повторная отправка приветственного уведомления: Если вы хотите повторно отправить приветственное уведомление, сначала убедитесь, что у вас есть активное уведомление, настроенное для события "User has been created (by administrator) (пользователь был создан администратором)". Затем выберите Admin menu > E-learning > Users. На главной странице Users выберите всех соответствующих пользователей, нажмите кнопку Choose action (выбрать действие) внизу страницы и выберите Send activation notification (отправить уведомление об активации).
  • Запись пользователей в неопубликованные курсы: При записи пользователей в курсы, которые еще не опубликованы, заранее решите, отправлять ли уведомления об этих записях, и обязательно отключите любые уведомления о записи перед записью пользователей. После завершения записи вы можете повторно активировать уведомления. Это может произойти, когда вам нужно записать пользователей в неопубликованные курсы, чтобы сохранить историю записей.
  • ICS файлы: ICS файлы, открытые в Outlook, откроются в новом календаре, а не в календаре по умолчанию.
